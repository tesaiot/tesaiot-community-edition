# nginx â€” serverTLS & mTLS front (TESAIoT Community Edition)

Apache-2.0. Single-org, self-host. nginx terminates TLS for the browser-facing
Admin UI + REST API (serverTLS) and provides a dedicated mTLS endpoint for IoT
devices.

> **Rendered file:** `conf.d/30-iot-mtls.conf` is generated by
> `scripts/generate-secrets.sh` from the committed `30-iot-mtls.conf.tpl`
> (the `X-MTLS-Gateway` marker from `.env` baked in; the rendered path is
> `.gitignore`d). Edit the `.tpl`, then re-render with `make secrets`.
> nginx only globs `*.conf`, so the `.tpl` itself is never loaded.

## Files

| File | Purpose |
|------|---------|
| `nginx.conf` | Main config: http context, rate-limit zones, Docker resolver, `include conf.d/*.conf`. |
| `conf.d/10-redirect.conf` | Port 80 â†’ HTTPS redirect (+ optional ACME challenge path). |
| `conf.d/20-admin-api.conf` | **serverTLS** on 443: Admin UI + `/api` (no client cert). |
| `conf.d/30-iot-mtls.conf` | **mTLS** on 9444: device telemetry ingest with `ssl_verify_client`. |

## Required certificates

Mount your certs into the container at `/etc/nginx/certs/`:

| Path | Description | Used by |
|------|-------------|---------|
| `/etc/nginx/certs/server-cert.pem` | Server certificate (leaf + chain). | 443, 9444 |
| `/etc/nginx/certs/server-key.pem`  | Server private key. | 443, 9444 |
| `/etc/nginx/certs/ca-bundle.pem`   | Client CA bundle â€” the Vault PKI issuing-CA chain that signed device certs. | 9444 (mTLS) |

These are **not shipped** (no secrets in the distribution). Provide them via one
of:

- **Vault PKI** (recommended, in-scope): issue a server cert from the
  intermediate (`pki-int`) and export the CA chain as `ca-bundle.pem`. See the
  Vault PKI / vault-agent area.
- **Let's Encrypt**: point `server-cert.pem` / `server-key.pem` at
  `fullchain.pem` / `privkey.pem`; the `10-redirect.conf` already exposes the
  `/.well-known/acme-challenge/` path.
- **Self-signed (dev only)**:

  ```sh
  mkdir -p certs
  openssl req -x509 -newkey rsa:2048 -nodes -days 365 \
    -keyout certs/server-key.pem -out certs/server-cert.pem \
    -subj "/CN=localhost"
  # For mTLS, ca-bundle.pem must be the CA that signed your device client certs.
  ```

## Mode selection (mTLS vs serverTLS)

In `conf.d/30-iot-mtls.conf`:

- **mTLS, mixed fleet (default):** `ssl_verify_client optional;` â€” the API
  decides based on the forwarded `X-Client-Verify` / `X-Client-*` headers.
- **mTLS, enforced:** `ssl_verify_client on;` â€” nginx rejects any device without
  a valid client cert.
- **serverTLS only:** comment out both `ssl_client_certificate` and
  `ssl_verify_client`.

## Compose wiring (reference)

```yaml
nginx:
  image: nginx:1.27.5-alpine3.21
  volumes:
    - ./config/nginx/nginx.conf:/etc/nginx/nginx.conf:ro
    - ./config/nginx/conf.d:/etc/nginx/conf.d:ro
    - ./config/tls:/etc/nginx/certs:ro     # server-cert.pem, server-key.pem, ca-bundle.pem
  ports:
    - "80:80"
    - "443:443"
    - "9444:9444"   # IoT device mTLS ingest
  depends_on:
    - tesa-api
    - tesa-admin-ui
  healthcheck:
    test: ["CMD", "nginx", "-t"]
```

## Forwarded mTLS headers

The mTLS server block forwards these to the API for per-device authorization:
`X-Client-Cert` (URL-escaped PEM), `X-Client-Verify`, `X-Client-S-DN`,
`X-Client-I-DN`, `X-Client-Serial`, `X-Client-Fingerprint`.

> No real hostnames, IPs, or secrets are baked in. `server_name _;` matches any
> host; set a concrete FQDN for strict SNI matching. Replace any `CHANGEME` /
> placeholder values before production use.
