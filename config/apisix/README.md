# APISIX API Gateway (TESAIoT Community Edition)

Single-organization, Apache-2.0 distribution. APISIX runs in **standalone YAML
mode** (no etcd): all routes/SSL/consumers live in `apisix.yaml` and are loaded
at boot.

> **Rendered files:** `config.yaml` and `apisix.yaml` are generated by
> `scripts/generate-secrets.sh` from the committed `config.yaml.tpl` /
> `apisix.yaml.tpl` templates (admin key, sample consumer key and serverTLS
> material baked in from `.env`; rendered paths are `.gitignore`d). Edit the
> `.tpl`, then re-render with `make secrets`.

## Files

| File | Mounts to | Purpose |
|------|-----------|---------|
| `config.yaml` | `/usr/local/apisix/conf/config.yaml` | Runtime config: admin API (9180), HTTP (9080), HTTPS/serverTLS (9443), plugin allowlist, optional MQTT stream proxy. |
| `apisix.yaml` | `/usr/local/apisix/conf/apisix.yaml` | Declarative routes, server TLS cert (serverTLS mode), example device-key consumers. |
| `mtls-routes.yaml` | (not mounted) | Optional mTLS route patterns to merge into `apisix.yaml` when enabling client-cert auth. |

## Routes

| Route | URI | Upstream | Auth |
|-------|-----|----------|------|
| `device-telemetry-ip` | `POST /api/v1/telemetry` | `tesa-api:5566` | API key (`X-API-Key`) |
| `device-telemetry-device-id` | `POST /api/v1/devices/*/telemetry` | `tesa-api:5566` | API key (`X-API-Key`) |
| `api-backend` | `/api/*` | `tesa-api:5566` | JWT (enforced by backend) |
| `admin-ui` | `/*` (catch-all) | `tesa-admin-ui:80` | none (SPA + login) |

Route matching is most-specific-first, so the `/api/*` and telemetry routes win
over the `/*` admin-ui catch-all.

## Rate limiting (what is actually enforced)

> **Be precise about this; it is a security claim.**

The CE enforces **route-level** rate limiting only. The telemetry routes carry a
`limit-req` plugin (`rate`/`burst`, `rejected_code: 429`). Because APISIX runs in
**standalone YAML mode** (see below), there is **no runtime registration of a
per-device APISIX consumer**, so this `limit-req` is shared by all callers of the
route rather than being a true per-device gateway quota. The `key: consumer_name`
field only differentiates the handful of consumers statically declared in
`apisix.yaml`; with a single sample consumer it behaves as a route-wide limit.

Per-key controls that *are* genuinely per-device are enforced by the **API
backend** (`api_key_service.py`), not by the gateway.

There is **no** dynamic "one APISIX consumer per device" mechanism in the CE.
The old `apisix_consumer_service.py` that did this required an **etcd-backed**
APISIX Admin API; it is **disabled/dead code** in the CE and raises on use. To
get true per-device gateway rate limiting you must run APISIX with an etcd
control plane (out of scope for the single-org CE) and wire consumer
create/revoke into the device-key lifecycle.

## MQTT TCP stream proxy (optional)

EMQX terminates MQTT directly in the default topology, so the stream proxy is
**disabled** to avoid host-port clashes. To front MQTT through APISIX:

1. Uncomment the `stream_proxy.tcp` block in `config.yaml` (port `1884`).
2. Publish `1884` in `docker-compose.yml` for the apisix service.
3. Point devices at APISIX:1884; APISIX forwards to EMQX:1883 via the
   `mqtt-proxy` stream plugin.

## TLS modes

- **serverTLS**: paste a Vault-PKI server cert/key into the `ssls` block of
  `apisix.yaml` (port 9443). This is the default.
- **mTLS**: add a `client` block (CA + `verify_client: true`) to the `ssls`
  entry and merge `mtls-routes.yaml`. The Vault PKI area emits the client-CA
  bundle (`pki-int` ca_chain).

## Secrets

All secrets are placeholders and **must** be replaced before use:

- `config.yaml` -> `admin_key.key`: replace `CHANGEME_APISIX_ADMIN_KEY`
  (`openssl rand -hex 16`). The API backend reads it via `APISIX_ADMIN_KEY`.
- `apisix.yaml` -> `ssls`: replace the PLACEHOLDER cert/key with Vault-PKI material.
- `apisix.yaml` -> `consumers`: rotate `CHANGEME_DEVICE_API_KEY_SAMPLE`. This is
  a single static sample consumer. In standalone YAML mode the CE does **not**
  create per-device consumers dynamically (the Admin API cannot mutate runtime
  state and `apisix.yaml` is mounted read-only); device API keys are issued and
  validated by the API backend (`api_key_service.py`), not as APISIX consumers.

## Notes

- Out-of-scope routes from the original platform (WebSocket B2B `/ws/*`) were
  removed — B2B/WebSocket is not part of the Community Edition.
- etcd is not used; APISIX is a `data_plane` with `config_provider: yaml`.
