# EMQX MQTT Broker — TESAIoT Community Edition

Single-organization EMQX 5.x configuration for the Community Edition.
Image: `emqx/emqx:5.10.0`.

> **Rendered files:** `emqx.conf` and `auth-built-in-db-bootstrap.csv` are
> generated by `scripts/generate-secrets.sh` from the committed
> `emqx.conf.tpl` / `auth-built-in-db-bootstrap.csv.tpl` templates (secrets
> from `.env` baked in; rendered paths are `.gitignore`d). Edit the `.tpl`,
> then re-render with `make secrets`.

## Files

| File | Purpose |
|------|---------|
| `emqx.conf` | Authoritative broker config: listeners, auth chain, authorization, MQTT/log settings. |
| `acl.conf` | Optional file-based ACL **fallback** (NOT wired in by default; HTTP ACL is primary). |
| `auth-built-in-db-bootstrap.csv` | Seeds the internal `mqtt-bridge` service account into the built-in DB. |

## Listeners

| Port | Type | Auth | Use |
|------|------|------|-----|
| 1883 | TCP (plain) | webhook / built-in DB | local/dev only |
| 8083 | WS (plain) | webhook | local/dev only |
| 8883 | **mTLS** | client certificate (Vault PKI) + webhook | production devices |
| 8884 | **serverTLS** | username/password over TLS | production devices |
| 8084 | WSS (server TLS) | webhook | browser/dashboard |
| 18083 | Dashboard (HTTP) | dashboard login | admin |

- **mTLS (8883):** `verify_peer` + `fail_if_no_peer_cert = true`. The client must
  present a Vault-PKI-issued certificate; the API webhook additionally validates
  the certificate CN/serial against the device registry.
- **serverTLS (8884):** `verify_none`; the device authenticates with
  username/password (bcrypt `password_hash`) verified by the API webhook.

## Authentication & Authorization

Two-stage authentication chain in `emqx.conf`:

1. **built_in_database** — only the internal `mqtt-bridge` account
   (bootstrapped from `auth-built-in-db-bootstrap.csv`). Always available, even
   while the API is restarting.
2. **HTTP webhook** — `POST http://tesa-api:5566/api/v1/emqx/auth` for all
   devices/users.

Authorization is a single HTTP source:
`POST http://tesa-api:5566/api/v1/emqx/acl` (deny on no-match, cache 1m).

Both webhooks send `Authorization: Bearer <secret>` which must match the API's
`EMQX_WEBHOOK_SECRET`.

## Certificates (Vault PKI)

The server cert/key and CA bundle are rendered by the Vault Agent sidecar into
the mounted `/opt/emqx/etc/certs` directory:

- `cert-with-chain.pem` — server certificate + intermediate chain
- `key.pem` — server private key
- `vault-ca-bundle.pem` — trust anchor for verifying device client certs (mTLS)

These files are auto-renewed by the Vault Agent; EMQX reloads on rotation. The
broker will not start its TLS listeners until these files exist.

## Required secrets — replace every `CHANGEME_*`

Set these via environment variables in `docker-compose.yml` / `.env` (preferred)
or by editing the config. **Never commit real values.**

| Placeholder | Env override | Meaning |
|-------------|--------------|---------|
| `CHANGEME_EMQX_COOKIE` | `EMQX_NODE__COOKIE` | cluster cookie |
| `CHANGEME_DASHBOARD_PASSWORD` | `EMQX_DASHBOARD__DEFAULT_PASSWORD` | dashboard admin password |
| `CHANGEME_WEBHOOK_SECRET` | `EMQX_WEBHOOK_SECRET` (must match API) | auth/ACL webhook bearer token |
| `CHANGEME_BRIDGE_MQTT_PASSWORD` | `MQTT_BRIDGE_PASSWORD` | internal mqtt-bridge account password |

> The webhook bearer secret and the mqtt-bridge password are referenced in two
> places each (broker side + API/bridge side) — keep them in sync via `.env`.

## Compose mounts (reference)

```yaml
emqx:
  image: emqx/emqx:5.10.0
  ports: ["1883:1883","8883:8883","8884:8884","8084:8084","18083:18083"]
  volumes:
    - ./config/emqx/emqx.conf:/opt/emqx/etc/emqx.conf:ro
    - ./config/emqx/auth-built-in-db-bootstrap.csv:/opt/emqx/etc/auth-built-in-db-bootstrap.csv:ro
    - ./config/certificates/certs/emqx:/opt/emqx/etc/certs:ro   # Vault-rendered
  depends_on:
    api: { condition: service_healthy }
```

## Excluded vs. the full platform

Dropped as out-of-scope for CE: MQTT-over-QUIC listener (`emqx-quic.conf`),
the Rust `vault-auth-service` auth fragments (`auth.conf`,
`authentication.conf`, `authorization.conf`), Prometheus exporter
(`prometheus.conf`), dashboard API keys (`api_keys.txt`), the production PKI env
files, OTA/CSR/protected-update ACL rules, B2B MQTT API-token paths, the
analytics-service ACL grant, and multi-org topic isolation.
