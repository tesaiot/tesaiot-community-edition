# การปฏิบัติตามมาตรฐาน: ETSI EN 303 645 และแนวปฏิบัติความมั่นคงปลอดภัยไซเบอร์ IoT ของ สกมช.

เอกสารนี้อธิบายว่า TESAIoT เกี่ยวข้องกับมาตรฐานความมั่นคงปลอดภัยไซเบอร์สำหรับ IoT ผู้ใช้ทั่วไปสองฉบับอย่างไร เขียนขึ้นสำหรับวิศวกรที่ต้องตัดสินใจจัดซื้อหรือนำไปใช้งานจริง

มาตรฐานสองฉบับคือ

* ETSI EN 303 645 V2.1.1 หัวข้อ Cyber Security for Consumer Internet of Things: Baseline Requirements กำหนดข้อกำหนดหลัก 13 หมวด พร้อมหมวดคุ้มครองข้อมูลส่วนบุคคล
* แนวปฏิบัติการรักษาความมั่นคงปลอดภัยไซเบอร์สำหรับอุปกรณ์ IoT สำหรับผู้ใช้งานทั่วไป v1.0 ของ สกมช. กำหนดข้อ 6.1 ถึง 6.26 จัดเป็นสี่ระดับแบบสะสม พัฒนาต่อยอดจาก ETSI EN 303 645 ร่วมกับ ISO/IEC 27400, 27402, 29147 และ 30111

ก่อนถึงตารางผลลัพธ์ มีจุดหนึ่งที่กำหนดวิธีอ่านทั้งหมด มาตรฐานทั้งสองเขียนขึ้นสำหรับตัวอุปกรณ์และผู้ผลิตเป็นหลัก แต่ TESAIoT คือแพลตฟอร์มที่อุปกรณ์เหล่านั้นเชื่อมต่อเข้ามา ได้แก่ API, MQTT broker, ระบบออกใบรับรอง และคอนโซลบริหารจัดการ แพลตฟอร์มทำตามครึ่งฝั่งเซิร์ฟเวอร์ของข้อกำหนดได้หลายข้อโดยตรง ส่วนครึ่งฝั่งอุปกรณ์ ได้แก่ secure boot, secure element บนตัวอุปกรณ์, debug port และการเปิดเผยข้อมูลเซนเซอร์ เป็นของเฟิร์มแวร์อุปกรณ์ และยังมีอีกกลุ่มที่เป็นเรื่องเชิงปฏิบัติการ คือขึ้นกับวิธีดูแลระบบเมื่อเวลาผ่านไป เช่น การอัปเดตให้ทันเวลา การเฝ้าระวัง และการว่าจ้างห้องปฏิบัติการที่ได้รับการรับรองมาทดสอบเจาะระบบ ด้วยเหตุนี้เราจึงอธิบายความครอบคลุมเป็นสามชั้น และระบุช่องว่างด้วยความแม่นยำเท่ากับที่อธิบายจุดแข็ง

## สามชั้นของความครอบคลุม

ชั้นแรกคือตัวแพลตฟอร์มเอง ฝั่งเซิร์ฟเวอร์ ซึ่งเป็นของซอฟต์แวร์ TESAIoT ทำไว้ในซอร์สโค้ดโอเพนซอร์ส ได้แก่ ระบบ PKI, mutual TLS, การยืนยันตัวตนแบบ fail closed, การจำกัดอัตราและการล็อกบัญชี, การเก็บความลับอย่างปลอดภัย, การลดพื้นผิวการโจมตี, การตรวจสอบข้อมูลนำเข้า, นโยบายเปิดเผยช่องโหว่ และการสแกน supply chain ใน CI

ชั้นที่สองคืออุปกรณ์อ้างอิง ซึ่งเป็นของเฟิร์มแวร์และฮาร์ดแวร์ของอุปกรณ์ TESA รองรับชั้นนี้ด้วยการออกแบบอุปกรณ์ปลอดภัยอ้างอิงที่ใช้ hardware secure element (Infineon OPTIGA Trust M) สำหรับ identity และการเก็บกุญแจที่ยึดกับฮาร์ดแวร์ และใช้ MCU ที่รองรับ secure boot (PSoC Edge E84 ที่มี Arm Cortex-M33 secure core) แพลตฟอร์มออกใบรับรองให้อุปกรณ์เหล่านี้อยู่แล้วในวันนี้ คือออก identity X.509 ที่ไม่ซ้ำต่ออุปกรณ์ผ่าน Vault PKI และรองรับเส้นทางใบรับรองของ OPTIGA Trust M โดย TESA ส่งมอบฝั่งอุปกรณ์เป็นแบบอ้างอิงและการอบรม

ชั้นที่สามคือการปฏิบัติการ ซึ่งเป็นของผู้ที่ดูแลระบบ ครอบคลุมการอัปเดตให้ทันเวลา การเฝ้าระวัง การประกาศช่วงเวลาสนับสนุน และการจัดให้มีการทดสอบเจาะระบบโดยห้องปฏิบัติการที่ได้รับการรับรอง สำหรับ Community Edition ที่คุณ self host เอง คุณเป็นเจ้าของชั้นนี้ สำหรับ Enterprise Cloud ที่ TESA ดูแลให้ TESA เป็นเจ้าของชั้นนี้

## ผลลัพธ์

สำหรับ Enterprise Cloud ที่ TESA บริหารจัดการเต็มรูปแบบ บริการถูกออกแบบมาให้เป็นไปตาม ETSI EN 303 645 และครบทั้งสี่ระดับของ สกมช. แบบครบวงจร TESA เป็นเจ้าของชั้นปฏิบัติการ ได้แก่ การอัปเดตที่เซ็นและทันเวลา การเฝ้าระวัง ความยืดหยุ่น การทดสอบเจาะระบบโดยห้องปฏิบัติการที่ได้รับการรับรองเป็นระยะ และวงรอบการสนับสนุนที่เผยแพร่ และจับคู่แพลตฟอร์มที่บริหารจัดการกับอุปกรณ์ปลอดภัยอ้างอิงของ TESA จึงครอบคลุมข้อกำหนดฝั่งอุปกรณ์ด้วย

สำหรับ Community Edition ที่ self host ภายใต้ Apache-2.0 แพลตฟอร์มทำตาม baseline ระดับ 1 และระดับ 2 ของ สกมช. ได้ประมาณ 70 เปอร์เซ็นต์ทันทีที่ชั้นแพลตฟอร์ม ความครอบคลุมจะเพิ่มเข้าใกล้เต็มเมื่อจับคู่ Community Edition กับอุปกรณ์ปลอดภัยอ้างอิงของ TESA (ชั้นที่สอง) และกับการปฏิบัติการที่ดี (ชั้นที่สาม) ช่องว่างของผลิตภัณฑ์ที่เหลือระบุไว้ด้านล่างและติดตามไว้ใน roadmap

ความต่างระหว่างสองรุ่นไม่ได้มาจากโค้ดที่อ่อนแอกว่า ทั้งสองรุ่นรันแกนความปลอดภัยชุดเดียวกัน ความต่างคือบริการที่บริหารจัดการให้สามารถรับประกันชั้นปฏิบัติการและส่งมอบอุปกรณ์ที่ผ่านการรับรองได้ด้วย ส่วนการติดตั้งแบบ self host ผลลัพธ์สุดท้ายขึ้นกับวิธีที่ผู้ดูแลนำไปใช้งาน

## สิ่งที่แพลตฟอร์มทำได้ทันที

ข้อความด้านล่างตรวจสอบกับซอร์สโค้ดจริง ณ รุ่น v1.0.0

### ระดับ 1 ฐานความมั่นคงปลอดภัย

ไม่มีรหัสผ่านเริ่มต้นแบบสากล (ข้อ 6.1.1 และ ETSI 5.1) ทุกการติดตั้งสร้าง secret สุ่มที่แข็งแรง และแพลตฟอร์มจะไม่ยอมเริ่มทำงานใน production หาก secret ใดยังเป็นค่า placeholder หรืออ่อนหรือสั้นเกินไป ดู `scripts/generate-secrets.sh` และ `services/api/api/core/config.py` อุปกรณ์ได้รับใบรับรอง X.509 ที่ไม่ซ้ำต่ออุปกรณ์จาก Vault PKI ไม่ใช่ค่า default ที่ใช้ร่วมกัน

การยืนยันตัวตนและการต้านการเดารหัส (ข้อ 6.1.3 ถึง 6.1.5 และ ETSI 5.1) รหัสผ่านถูกแฮชด้วย bcrypt โดยปรับรอบได้ผ่าน environment การล็อกอินมีการล็อกทั้งราย IP และรายบัญชี (ใช้ Redis พร้อม fallback แบบ in memory) ทุกการตอบกลับ HTTP 429 มี header Retry-After และการเทียบ secret เป็นแบบ constant time ดู `services/api/api/utils/validation.py` และ `services/api/api/core/auth.py`

การเปิดเผยช่องโหว่ (ข้อ 6.2.1 และ ETSI 5.2) มีนโยบาย coordinated disclosure ฉบับเต็มใน `SECURITY.md` ทั้งภาษาอังกฤษและภาษาไทย

ช่วงเวลาสนับสนุนที่เผยแพร่และเวอร์ชันที่ระบุได้ (ข้อ 6.3.6 และ 6.3.7) ตาราง supported versions อยู่ใน `SECURITY.md` และระบุเวอร์ชันที่กำลังรันได้ผ่าน `VERSION.txt`, `CHANGELOG.md` และ header `X-Build-Fingerprint`

### ระดับ 2 สอดคล้องกับมาตรฐานสากล

การเก็บพารามิเตอร์ความปลอดภัยอย่างปลอดภัยและไม่ฝัง secret ในซอร์ส (ข้อ 6.4.1 และ 6.4.3 และ ETSI 5.4) กุญแจส่วนตัวของอุปกรณ์อยู่ใน HashiCorp Vault เท่านั้น ส่วน MongoDB เก็บเพียงตัวอ้างอิง secret ทั้งหมดมาจากไฟล์ environment (ตั้งสิทธิ์เป็น 600) หรือจาก Vault ไฟล์คอนฟิกที่ commit เป็นเทมเพลตที่มีแต่ค่า placeholder และไฟล์ที่ render แล้วถูก git ignore การสแกน secret ใน CI บังคับเรื่องนี้ และแพลตฟอร์มจะไม่เริ่มทำงานหากยังมี placeholder ใน production

กุญแจไม่ซ้ำต่ออุปกรณ์ (ข้อ 6.4.4) Vault PKI เป็นผู้ออกใบรับรอง และ common name ถูกบังคับฝั่งเซิร์ฟเวอร์ให้เป็น device id อุปกรณ์จึงขอ identity ของอุปกรณ์อื่นไม่ได้

สื่อสารอย่างปลอดภัย (ข้อ 6.5.1 ถึง 6.5.4 และ ETSI 5.5) แพลตฟอร์มใช้ TLS 1.2 และ 1.3 พร้อมรายการ cipher แบบ AEAD เท่านั้น (ไม่มี CBC ไม่มี SHA-1) ให้บริการ MQTT ผ่าน TLS เท่านั้น listener แบบ mutual TLS ของ EMQX บังคับให้มีใบรับรองฝั่งไคลเอนต์ ปฏิเสธหากไม่มี และตรวจรายการเพิกถอนใบรับรองระหว่าง handshake ตัว gateway marker ของ mutual TLS ปลอมไม่ได้ เทียบแบบ constant time และ fail closed กุญแจส่วนตัวของอุปกรณ์ถูกส่งมอบครั้งเดียวโดยเข้ารหัสแบบไฮบริด (RSA-OAEP หรือ ECDH ร่วมกับ AES-256-GCM) และวงจรกุญแจทั้งหมด คือออก ต่ออายุ และเพิกถอน ทำงานใน Vault

ลดพื้นผิวการโจมตี (ข้อ 6.6.1 ถึง 6.6.3 และ ETSI 5.6) API และฐานข้อมูลไม่ถูก publish ออกสู่ host พอร์ต management และพอร์ตที่ไม่เข้ารหัสผูกกับ 127.0.0.1 เท่านั้น เครือข่าย Docker ถูกแบ่งแยก คอนเทนเนอร์ bridge รันแบบ no new privileges แบบ read only และตัดสิทธิ์ทั้งหมด API รันด้วย gunicorn ใน production โดยไม่มี debug server

คุ้มครองข้อมูลส่วนบุคคลระหว่างส่ง (ข้อ 6.7.1 และ ETSI 5.8) ทราฟฟิกของอุปกรณ์และผู้ดูแลทั้งหมดใช้ TLS 1.2 หรือ 1.3

ตรวจสอบข้อมูลนำเข้า (ข้อ 6.9.1 และ ETSI 5.13) แพลตฟอร์มตรวจ schema บังคับขนาด request ตรวจ device id และ telemetry ป้องกัน URL webhook ขาออกจาก server side request forgery และตั้ง security header ในทุกการตอบกลับ

### ระดับ 3 ความมั่นคงปลอดภัยตั้งแต่การออกแบบ (สอดคล้องกับ ISO/IEC 27400)

แพลตฟอร์มทำตามข้อกำหนดด้านวิศวกรรมและกระบวนการที่ใช้กับผลิตภัณฑ์ซอฟต์แวร์ ได้แก่ ไปป์ไลน์การพัฒนาที่ปลอดภัย (CI รัน shellcheck, ruff, type check และ build, การตรวจ compose และการสแกน dependency, secret และ image) การดูแล dependency ให้ทันสมัย (CI ล้มเมื่อพบระดับ HIGH หรือ CRITICAL) การ hardening ก่อนปล่อย การสื่อสารด้านความปลอดภัยที่ชัดเจนผ่าน `CHANGELOG.md` และ `SECURITY.md` และกระบวนการจัดการช่องโหว่ที่กำหนดไว้ มีสองข้อที่ทำได้บางส่วน คือ software bill of materials สร้างให้ image ของ admin UI แล้วแต่ยังไม่ครบทุก image และการประเมินช่องโหว่ปัจจุบันยังเป็นแบบอัตโนมัติเท่านั้น

## ช่องว่างใน Community Edition ที่ติดตามไว้ใน roadmap

เราระบุไว้ตรงไปตรงมาเพื่อให้วิศวกรวางแผนรับมือได้

1. ส่วนติดต่อด้านความยินยอมและความเป็นส่วนตัว (ข้อ 6.10.1 ถึง 6.10.4 และ ETSI clause 6) แพลตฟอร์มยังไม่มีส่วนติดต่อสำหรับเก็บความยินยอม นโยบายความเป็นส่วนตัว หรือการเปิดเผย telemetry การใช้งานที่อยู่ใต้กฎหมายต้องเพิ่มส่วนนี้เอง การ self host ช่วยได้ เพราะ telemetry ทั้งหมดอยู่บนโครงสร้างพื้นฐานของคุณเอง
2. การลบข้อมูลเกินระดับบัญชี (ข้อ 6.8.1 และ ETSI 5.11) มีการลบบัญชีผู้ใช้ ซึ่งลบข้อมูลทั้งจาก MongoDB และ Vault แต่ยังไม่มีการลบหรือทำให้ไม่ระบุตัวตนของ telemetry แบบอัตโนมัติ และยังไม่มีฟีเจอร์ export ข้อมูล
3. การเซ็นอัปเดตเชิงรหัสลับ (ข้อ 6.3.5 และ ETSI 5.7) image ถูก pin ด้วย version tag และสแกนใน CI แต่ยังไม่ pin ด้วย SHA digest หรือเซ็น เช่นด้วย cosign ส่วน secure boot และ software integrity (ETSI 5.7) ทำที่ชั้นอุปกรณ์ผ่าน secure boot ของ PSoC Edge
4. เอกสาร threat model อย่างเป็นทางการ (ข้อ 6.11) เหตุผลเชิงภัยคุกคามมีอยู่ในโค้ดและคอนฟิกแล้ว แต่ยังไม่มีเอกสาร STRIDE แยกฉบับ
5. การทดสอบเจาะระบบแบบกล่องดำโดยห้องปฏิบัติการที่ได้รับการรับรอง (สกมช. ระดับ 4 ข้อ 6.19 ถึง 6.26) ไม่รวมในชุดโอเพนซอร์ส มีให้ผ่านเส้นทางการรับรองของ TESA หรือการทำงานแบบ Enterprise
6. ข้อเชิงปฏิบัติการ (ข้อ 6.3.1 ถึง 6.3.4 คือการอัปเดตทันเวลา การสนับสนุน และการเฝ้าระวัง) ผู้ดูแล self host เป็นเจ้าของใน Community Edition ส่วน TESA เป็นเจ้าของใน Enterprise Cloud
7. ข้อที่เป็นของอุปกรณ์เท่านั้น (secure element ที่ต้านการงัดแงะ ข้อ 6.4.2 และ ETSI 5.4, การปิด debug port บนอุปกรณ์ ข้อ 6.6.3, การเปิดเผยเซนเซอร์ ข้อ 6.7.2 และ secure boot ETSI 5.7) ข้อเหล่านี้ทำได้ด้วยอุปกรณ์ปลอดภัยอ้างอิงของ TESA ซึ่งเป็นชั้นที่สอง ไม่ใช่แพลตฟอร์มลำพัง

## หมายเหตุเรื่องข้อ ETSI EN 303 645 ที่ สกมช. ยังไม่นำมาใช้

ภาคผนวก ค ของ สกมช. v1.0 ระบุข้อของ ETSI EN 303 645 หลายข้อที่ยังไม่นำมาใช้ ในจำนวนนั้นมีสามวัตถุประสงค์เต็มหมวดที่ไม่มีข้อเทียบเท่าใน สกมช. เลย และ TESAIoT จึงติดตามแยกต่างหาก ได้แก่ 5.7 (ความครบถ้วนของซอฟต์แวร์ รวมถึง secure boot) 5.9 (ความยืดหยุ่นต่อการขาดการเชื่อมต่อ) และ 5.10 (การตรวจ telemetry หาความผิดปกติ) อุปกรณ์อ้างอิงของ TESA ครอบคลุม 5.7 ผ่าน secure boot ของ PSoC Edge และครอบคลุม 5.4 ผ่าน secure element ของ OPTIGA Trust M ส่วน 5.9 และ 5.10 เป็นส่วนหนึ่งของการปฏิบัติการใน Enterprise Cloud ที่บริหารจัดการให้ ดังนั้นคำว่า ทำตาม สกมช. กับ ทำตาม ETSI EN 303 645 เต็มรูปแบบ เป็นคนละข้อความกัน และเอกสารนี้แยกทั้งสองออกจากกัน
