# การปฏิบัติตามมาตรฐาน — ETSI EN 303 645 และแนวปฏิบัติความมั่นคงปลอดภัยไซเบอร์ IoT ของ สกมช. (NCSA)

เอกสารนี้ทำการแมป **TESAIoT** กับมาตรฐานความมั่นคงปลอดภัยไซเบอร์สำหรับ IoT ผู้ใช้งานทั่วไป 2 ฉบับ:

- **ETSI EN 303 645 V2.1.1** — *Cyber Security for Consumer Internet of Things: Baseline Requirements* (ข้อกำหนดหลัก 13 หมวด + หมวดคุ้มครองข้อมูลส่วนบุคคล)
- **แนวปฏิบัติการรักษาความมั่นคงปลอดภัยไซเบอร์สำหรับอุปกรณ์ IoT สำหรับผู้ใช้งานทั่วไป v1.0** (สกมช.) — ข้อ **6.1–6.26** จัดเป็น **4 ระดับแบบสะสม** อิงตาม ETSI EN 303 645 และ ISO/IEC 27400/27402/29147/30111

> **อ่านอย่างตรงไปตรงมา** มาตรฐานทั้งสองเขียนขึ้นสำหรับ **ตัวอุปกรณ์ (device)** และ **ผู้ผลิต** เป็นหลัก แต่ TESAIoT คือ **แพลตฟอร์มฝั่งเซิร์ฟเวอร์** ที่อุปกรณ์เชื่อมต่อเข้ามา (API, MQTT broker, PKI, dashboard) แพลตฟอร์มทำได้โดยตรงเฉพาะ *ครึ่งฝั่ง backend* ของหลายข้อกำหนด ส่วน *ครึ่งฝั่งอุปกรณ์* (secure boot, secure element บนอุปกรณ์, debug port, การเปิดเผยเซนเซอร์) เป็นของอุปกรณ์ และอีกหลายข้อเป็นเรื่อง *เชิงปฏิบัติการ* (อัปเดตให้ทันเวลา, การเฝ้าระวัง, การทดสอบเจาะระบบโดยห้องปฏิบัติการ) เราจึงอธิบายความครอบคลุมเป็น **3 ชั้น** และเปิดเผย **ช่องว่าง** อย่างตรงไปตรงมาเท่ากับที่อธิบายจุดแข็ง

## ความครอบคลุม 3 ชั้น

| ชั้น | ใครรับผิดชอบ | ใน TESAIoT |
|------|-------------|------------|
| **1. แพลตฟอร์ม (ฝั่งเซิร์ฟเวอร์)** | ซอฟต์แวร์ TESAIoT | ทำไว้ในซอร์สโค้ดโอเพนซอร์ส: PKI, mTLS, การยืนยันตัวตนแบบ fail-closed, การจำกัดอัตรา, การเก็บความลับอย่างปลอดภัย, การลดพื้นผิวการโจมตี, การตรวจสอบข้อมูลนำเข้า, การเปิดเผยช่องโหว่, การสแกน supply-chain |
| **2. อุปกรณ์อ้างอิง (reference device)** | เฟิร์มแวร์/ฮาร์ดแวร์ของอุปกรณ์ | ครอบคลุมด้วย **สถาปัตยกรรมอุปกรณ์ปลอดภัยอ้างอิงของ TESA**: **hardware secure element / HSM (Infineon OPTIGA™ Trust M)** สำหรับ identity และการเก็บกุญแจที่ยึดกับฮาร์ดแวร์ และ **MCU ที่รองรับ secure boot (PSoC™ Edge E84, Arm Cortex-M33 secure core)** แพลตฟอร์ม**ออกใบรับรองให้อุปกรณ์เหล่านี้อยู่แล้ว** (X.509 ต่ออุปกรณ์ผ่าน Vault PKI, เส้นทางใบรับรอง OPTIGA Trust M) TESA ส่งมอบฝั่งอุปกรณ์เป็น reference design และการอบรม |
| **3. การปฏิบัติการ (operations)** | ผู้ดูแลระบบ | การอัปเดตให้ทันเวลา, การเฝ้าระวัง, การกำหนดช่วงสนับสนุน, การทดสอบเจาะระบบโดยห้องปฏิบัติการ — **Community Edition (self-host) ผู้ดูแลเป็นเจ้าของชั้นนี้** ส่วน **Enterprise Cloud นั้น TESA เป็นเจ้าของให้** |

## ผลสรุป

| รุ่น | ความครอบคลุม ETSI EN 303 645 / NCSA |
|--------|---------------------------------|
| **TESAIoT Enterprise Cloud** (บริหารจัดการเต็มรูปแบบโดย TESA) | **มุ่งครบ 100% ทั้ง NCSA ระดับ 1–4** TESA เป็นเจ้าของชั้นปฏิบัติการ (อัปเดตที่เซ็นและทันเวลา, การเฝ้าระวัง telemetry/ความผิดปกติ, ความยืดหยุ่น, การทดสอบเจาะระบบเป็นระยะ, ช่วงสนับสนุนที่เผยแพร่) และจับคู่แพลตฟอร์มกับอุปกรณ์ปลอดภัยอ้างอิงของ TESA ทำให้ครอบคลุมฝั่งอุปกรณ์ด้วย |
| **TESAIoT Community Edition** (self-host, Apache-2.0) | **≈ 70% ของ baseline NCSA ระดับ 1–2 / ETSI EN 303 645 ทำได้ทันทีที่ชั้นแพลตฟอร์ม** ความครอบคลุมจะเพิ่มเข้าใกล้เต็มเมื่อจับคู่ CE กับอุปกรณ์ปลอดภัยอ้างอิงของ TESA (ชั้น 2) และแนวปฏิบัติที่ดีของผู้ดูแล (ชั้น 3) ช่องว่างที่เหลือของผลิตภัณฑ์อยู่ใน roadmap |

ความต่างนี้**ไม่ได้**มาจากโค้ดที่อ่อนแอกว่า — Enterprise Cloud และ Community Edition รัน **แกนความปลอดภัยชุดเดียวกัน** ความต่างคือบริการที่บริหารจัดการให้สามารถ*รับประกันชั้นปฏิบัติการ*และส่งมอบอุปกรณ์ที่ผ่านการรับรองได้ด้วย ในขณะที่ของ self-host ผลลัพธ์สุดท้ายขึ้นกับวิธีที่ผู้ดูแลนำไปใช้งาน

## สิ่งที่แพลตฟอร์มทำได้ทันที (ตรวจสอบแล้วพร้อมหลักฐาน)

**ระดับ 1 — Security baseline**

- **ไม่มีรหัสผ่านเริ่มต้นสากล (6.1.1 / EN 5.1)** — ทุกการติดตั้งสร้าง secret สุ่มที่แข็งแรง, boot แบบ **fail-closed** ปฏิเสธค่า `CHANGEME`/อ่อน/สั้นใน production; อุปกรณ์ได้ **ใบรับรอง X.509 ที่ไม่ซ้ำต่ออุปกรณ์** (Vault PKI) ไม่มีค่า default ร่วม
- **การยืนยันตัวตน & การต้านการเดารหัส (6.1.3–6.1.5 / EN 5.1)** — bcrypt (รอบปรับผ่าน env), ล็อกทั้งราย IP และรายบัญชี (Redis + fallback in-memory), ทุก `429` มี `Retry-After`; เทียบ secret แบบ constant-time
- **การเปิดเผยช่องโหว่ (6.2.1 / EN 5.2)** — นโยบาย coordinated disclosure ครบ (`SECURITY.md`, EN+TH)
- **ช่วงสนับสนุนที่เผยแพร่ & ระบุเวอร์ชันได้ (6.3.6 / 6.3.7)** — ตาราง supported-versions ใน `SECURITY.md`; `VERSION.txt`, `CHANGELOG.md`, และ header `X-Build-Fingerprint`

**ระดับ 2 — มาตรฐานสากล**

- **เก็บพารามิเตอร์อ่อนไหวอย่างปลอดภัย / ไม่ฝัง secret ในซอร์ส (6.4.1, 6.4.3 / EN 5.4)** — กุญแจส่วนตัวของอุปกรณ์อยู่ใน HashiCorp Vault **เท่านั้น**; secret ทั้งหมดมาจาก `.env` (chmod 600) หรือ Vault; config ที่ commit เป็นเทมเพลต `*.tpl` ที่มีแต่ `CHANGEME`; CI สแกน secret บังคับ
- **กุญแจไม่ซ้ำต่ออุปกรณ์ (6.4.4)** — Vault PKI โดย **บังคับ common name ฝั่งเซิร์ฟเวอร์** ให้เป็น device id อุปกรณ์จึงขอ identity ของอุปกรณ์อื่นไม่ได้
- **สื่อสารอย่างปลอดภัย (6.5.1–6.5.4 / EN 5.5)** — TLS 1.2/1.3 cipher แบบ **AEAD เท่านั้น** (ไม่มี CBC/SHA-1); MQTT ผ่าน TLS เท่านั้น; EMQX mTLS `verify_peer` + `fail_if_no_peer_cert` + **ตรวจ CRL ที่ TLS handshake**; mTLS gateway-marker ที่ปลอมไม่ได้ (`hmac.compare_digest`, fail-closed); ส่งกุญแจส่วนตัวแบบเข้ารหัสไฮบริดครั้งเดียว (RSA-OAEP/ECDH + AES-256-GCM); วงจรกุญแจครบใน Vault (ออก/ต่ออายุ/เพิกถอน)
- **ลดพื้นผิวการโจมตี (6.6.1–6.6.3 / EN 5.6)** — API และฐานข้อมูล **ไม่ publish** สู่ host; พอร์ต management/plaintext ผูก `127.0.0.1` เท่านั้น; แยก Docker network; `no-new-privileges`, read-only และ `cap_drop: ALL` ที่ bridge; รัน gunicorn ใน production (ไม่ใช่ debug server)
- **คุ้มครองข้อมูลส่วนบุคคลระหว่างส่ง (6.7.1 / EN 5.8)** — ทราฟฟิกอุปกรณ์และแอดมินผ่าน TLS 1.2/1.3
- **ตรวจสอบข้อมูลนำเข้า (6.9.1 / EN 5.13)** — ตรวจ schema, จำกัดขนาด request, ตรวจ device-id/telemetry, **กัน SSRF** ที่ URL webhook ขาออก, security headers ทุก response

**ระดับ 3 — Security-by-Design (ตาม ISO/IEC 27400)**

- **ทำแล้ว:** secure engineering/SDLC (CI รัน shellcheck, ruff, type-check + build, compose validate, **สแกน dependency, secret และ image**); dependency ทันสมัย (CI ล้มเมื่อเจอ HIGH/CRITICAL); hardening ก่อนปล่อย; สื่อสารความปลอดภัยชัด (`CHANGELOG.md`, `SECURITY.md`); การจัดการช่องโหว่
- **บางส่วน:** สร้าง SBOM ให้ admin-UI image (CycloneDX) แต่ยังไม่ครบทุก image; การประเมินช่องโหว่ยังเป็นแบบอัตโนมัติเท่านั้น

## ช่องว่างที่ตรงไปตรงมา (Community Edition, ติดตามใน roadmap)

เราอยากให้คุณเชื่อใจเพราะเราแม่นยำ ไม่ใช่เพราะปัดตัวเลขขึ้น

1. **UX ความยินยอม/ความเป็นส่วนตัว (6.10.1–6.10.4 / EN §6)** — แพลตฟอร์มยังไม่มี UI เก็บ consent, นโยบายความเป็นส่วนตัว หรือการเปิดเผย telemetry; การใช้งานที่อยู่ใต้กฎหมายต้องเพิ่มเอง *ข้อดีของ CE: self-host ทำให้ telemetry อยู่บนโครงสร้างพื้นฐานของคุณเอง*
2. **สิทธิ์ลบข้อมูลเกินระดับบัญชี (6.8.1 / EN 5.11)** — มีลบบัญชีผู้ใช้ (MongoDB + Vault) แต่ยังไม่มีการลบ/ทำให้ไม่ระบุตัวตนของ telemetry และการ export ข้อมูล
3. **การเซ็นอัปเดตเชิงรหัสลับ (6.3.5 / EN 5.7)** — image pin ด้วย tag + CI สแกน แต่ยังไม่ pin ด้วย SHA-digest หรือเซ็น (เช่น cosign); **secure boot / software integrity (EN 5.7)** ทำที่**ชั้นอุปกรณ์** (PSoC Edge secure boot)
4. **เอกสาร threat model อย่างเป็นทางการ (6.11)** — มีเหตุผลเชิงภัยคุกคามฝังในโค้ด/คอนฟิกแล้ว แต่ยังไม่มีเอกสาร STRIDE แยก
5. **การทดสอบเจาะระบบกล่องดำโดยห้องปฏิบัติการรับรอง (NCSA ระดับ 4, 6.19–6.26)** — ไม่รวมในชุดโอเพนซอร์ส; มีให้ผ่านเส้นทางการรับรองของ TESA / Enterprise
6. **ข้อเชิงปฏิบัติการ (6.3.1–6.3.4 อัปเดตทันเวลา, สนับสนุน, เฝ้าระวัง)** — ผู้ดูแล self-host เป็นเจ้าของใน CE; TESA เป็นเจ้าของใน Enterprise Cloud
7. **ข้อฝั่งอุปกรณ์เท่านั้น** (secure element ต้านการงัดแงะ 6.4.2/EN 5.4, ปิด debug port บนอุปกรณ์ 6.6.3, เปิดเผยเซนเซอร์ 6.7.2, secure boot EN 5.7) — ครอบคลุมด้วย**อุปกรณ์ปลอดภัยอ้างอิงของ TESA** (ชั้น 2) ไม่ใช่แพลตฟอร์มลำพัง

## หมายเหตุ: ข้อ EN 303 645 ที่ NCSA ยังไม่นำมาใช้

ภาคผนวก ค ของ NCSA v1.0 ระบุข้อของ EN 303 645 หลายข้อที่ยังไม่ได้นำมาใช้ ในจำนวนนั้นมี **3 หมวดหลัก** ที่ไม่มีคู่ใน NCSA เลย และ TESAIoT จึงติดตามแยกต่างหาก: **5.7 software integrity / secure boot**, **5.9 ความยืดหยุ่นต่อการขาดการเชื่อมต่อ**, **5.10 ตรวจ telemetry หาความผิดปกติ** อุปกรณ์อ้างอิงของ TESA ครอบคลุม **5.7** (PSoC Edge secure boot) และ **5.4** (OPTIGA Trust M secure element); ส่วน 5.9 และ 5.10 เป็นส่วนของการปฏิบัติการใน Enterprise Cloud ดังนั้น "ตรงตาม NCSA" กับ "ตรงตาม EN 303 645 เต็ม" จึงถูกแยกและรายงานตามจริงในเอกสารนี้
